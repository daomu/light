# 微光｜技术方案（Rust + ESP-IDF）

## 1. 技术目标

本方案用于实现一款插电式智能小夜灯，运行在 **ESP32-C3-Zero** 上，使用：

- **BH1750**：检测环境亮度
- **AM312**：检测人体经过
- **单个 5mm 暖白漫射 LED**：夜灯发光
- **Type-C 持续供电**
- **Rust + ESP-IDF**
- **FreeRTOS 轻量任务 + 事件队列**

实现目标：

1. 室内环境由亮变暗时，自动开灯
2. 开灯后以 **20 分钟**渐暗方式逐步熄灭
3. 灯亮期间，若环境重新变亮，则立即关灯
4. 灯熄灭后，如果环境仍暗，**不会因持续黑暗而重复自动开灯**
5. 灯熄灭后，如果环境仍暗且 **AM312 检测到人经过**，则重新开灯
6. 开发板自带的用户 LED 如可控，必须始终保持关闭；如为硬件常亮电源灯，则必须硬件去除或遮蔽

---

## 2. 技术选型

## 2.1 主控与框架

- 主控开发板：**ESP32-C3-Zero**
- 语言：**Rust**
- 平台框架：**ESP-IDF**
- 并发模型：**FreeRTOS 任务 + ISR + 事件队列**
- 日志：**串口日志**

## 2.2 选择 Rust + ESP-IDF 的原因

当前项目需要：

- GPIO 中断
- I2C 读取 BH1750
- PWM 控制 LED
- FreeRTOS 任务与定时
- 串口日志
- 较稳定的外围支持

相比 `esp-hal / no_std`，`Rust + ESP-IDF` 更适合当前项目的原因是：

1. **外设支持更成熟**
2. **FreeRTOS 任务与定时器更顺手**
3. **调试路径更稳定**
4. **更符合“成熟工具优先”的原则**
5. 仍可用 Rust 编写核心业务逻辑与状态机

---

## 3. 硬件方案

## 3.1 硬件组成

- ESP32-C3-Zero × 1
- BH1750 模块板 × 1
- AM312 模块 × 1
- 5mm 暖白漫射 LED × 1
- 限流电阻 × 1（默认 **220Ω**）
- 杜邦线 / 焊线若干

---

## 3.2 必须先确认的硬件前提

这部分非常关键。

### A. ESP32-C3-Zero 板上 LED
如果板上存在：

- **可编程用户 LED**：可在软件中强制关闭
- **直接接 5V / 3.3V 的电源指示灯**：**软件无法关闭**

如果你的板子有不可控的常亮电源灯，那么它会破坏“夜灯产品”的体验。此时必须做以下任一处理：

1. 拆除该 LED
2. 拆除其限流电阻
3. 做物理遮光
4. 更换开发板

### B. BH1750 模块板电源灯
很多 BH1750 模块板也可能带有电源指示灯。  
如果该灯常亮，也会影响夜间环境。处理原则同上：

- 能拆则拆
- 不能拆则遮光
- 最好选择无电源灯模块

### C. AM312 供电电压
**AM312 必须接 3.3V 供电，不要接 5V。**

原因：

- 这样它的输出高电平也会是 3.3V 级别
- 可直接安全接入 ESP32-C3 GPIO
- 避免 5V 输出损坏芯片 GPIO

---

## 3.3 LED 选型要求

本方案采用：

- **单个 5mm 暖白漫射 LED**
- 直接由 ESP32 GPIO 输出 PWM 驱动
- 串联限流电阻
- 不加三极管 / MOSFET

该方案成立的前提：

1. LED 目标亮度为**柔和夜灯亮度**
2. 使用的是**低电流也能发光明显**的 LED
3. 最好选：
   - 暖白
   - 漫射 / 雾面
   - 低电流高亮型号
   - 在 2mA~5mA 区间内仍能有较好表现

如果后续实测过暗，再单独升级驱动方案；当前 MVP 不考虑该分支。

---

## 4. 推荐引脚分配

以下引脚分配以 **ESP32-C3-Zero 常见可用引脚**为基础，优先避开：

- USB 相关引脚
- 易产生歧义的引脚
- 常见启动相关引脚
- 板载 LED / 特殊用途高风险引脚

## 4.1 推荐分配

| 模块 | 信号 | 推荐 GPIO | 说明 |
|---|---|---:|---|
| BH1750 | SDA | GPIO6 | I2C 数据线 |
| BH1750 | SCL | GPIO7 | I2C 时钟线 |
| AM312 | OUT | GPIO4 | 作为人体事件输入，中断触发 |
| LED | PWM 输出 | GPIO5 | 控制夜灯亮度 |

## 4.2 供电连接

| 模块 | 引脚 | 连接 |
|---|---|---|
| BH1750 | VCC | 3V3 |
| BH1750 | GND | GND |
| BH1750 | ADDR | GND（地址固定为 `0x23`） |
| AM312 | VCC | 3V3 |
| AM312 | GND | GND |
| LED | 阴极 | GND |
| LED | 阳极 | 通过电阻接 GPIO5 |

---

## 4.3 为什么不用 GPIO8 / GPIO9
本方案故意不优先使用 GPIO8 / GPIO9 作为 I2C 引脚，原因是：

- 某些 ESP32-C3 板型上这些引脚与启动配置 / 板载功能更容易产生干扰
- 不利于提高“开箱即用”的稳定性

因此本方案默认推荐：

- `GPIO6 -> SDA`
- `GPIO7 -> SCL`

- 必须串联电阻

## 6. 软件架构

本方案采用：

- **一个核心控制任务**
- **一个光照采样任务**
- **一个 AM312 GPIO ISR**
- **一个渐暗周期定时器**
- **一个事件队列**

整体逻辑不采用“纯大循环”，而是采用轻量事件驱动结构。

---

## 6.1 架构图

```text
                 ┌──────────────────────┐
                 │   AM312 GPIO ISR     │
                 │  仅投递人体事件      │
                 └─────────┬────────────┘
                           │
                           ▼
                    ┌────────────┐
                    │ 事件队列    │
                    └─────┬──────┘
                          │
      ┌───────────────────┼───────────────────┐
      │                   │                   │
      ▼                   ▼                   ▼
┌──────────────┐   ┌──────────────┐   ┌────────────────┐
│ 光照采样任务  │   │ 渐暗定时器    │   │ 控制任务         │
│ 每1秒读BH1750 │   │ 每5秒发Tick   │   │ 状态机/开关灯逻辑 │
└──────────────┘   └──────────────┘   └────────────────┘
                                              │
                                              ▼
                                       ┌────────────┐
                                       │ LED PWM 控制│
                                       └────────────┘
```

---

## 6.2 任务职责

### A. 光照采样任务 `light_sensor_task`
职责：

- 初始化 BH1750 为连续高分辨率模式
- 每 **1 秒**读取一次 lux
- 根据双阈值和确认规则，判断：
  - 是否进入暗环境
  - 是否退出暗环境
- 向事件队列投递：
  - `EnvDarkEntered`
  - `EnvBrightEntered`
  - 可选地记录当前 lux 到共享状态

### B. 控制任务 `controller_task`
职责：

- 维护系统状态机
- 接收所有事件
- 决定：
  - 开灯
  - 关灯
  - 启动 / 停止渐暗定时器
  - 忽略无效事件
- 维护内部标志：
  - 当前是否处于暗环境
  - 暗环境自动触发是否已被消费

### C. AM312 GPIO ISR
职责：

- 只做极轻量操作
- 记录“有人体事件发生”
- 向事件队列投递 `MotionDetected`
- 不在 ISR 中直接开灯/关灯

### D. 渐暗定时器 `fade_timer`
职责：

- 仅在灯亮期间启动
- 每 **5 秒**投递一次 `FadeTick`
- 共执行 **240 次**
- 第 240 次后触发熄灯

---

## 7. 状态机设计

## 7.1 主状态

只保留两个主状态：

```text
Idle      ：灯灭监听
Lighting  ：灯亮渐暗
```

---

## 7.2 关键内部标志

### `env_dark: bool`
表示当前环境是否已经稳定判定为“暗环境”。

### `auto_dark_trigger_armed: bool`
表示是否允许“亮 -> 暗”这次边沿自动开灯。

作用：

- 解决“持续黑暗不允许反复自动开灯”的问题

规则：

- 当环境重新变亮时，设置为 `true`
- 当一次亮->暗自动开灯被触发后，设置为 `false`
- 当因为人体事件在黑暗中开灯时，不改变其核心用途
- 在持续黑暗中，20 分钟结束后不会再次自动开灯

---

## 7.3 状态迁移规则

### Idle 状态

#### 事件 1：`EnvDarkEntered`
条件：

- 当前 `auto_dark_trigger_armed == true`

动作：

- 开灯
- 启动渐暗定时器
- 进入 `Lighting`
- 设置 `auto_dark_trigger_armed = false`

#### 事件 2：`MotionDetected`
条件：

- 当前 `env_dark == true`

动作：

- 开灯
- 启动渐暗定时器
- 进入 `Lighting`

#### 事件 3：`EnvBrightEntered`
动作：

- 设置 `env_dark = false`
- 设置 `auto_dark_trigger_armed = true`
- 保持 `Idle`

---

### Lighting 状态

#### 事件 1：`FadeTick`
动作：

- 亮度按查表方式下降一级
- 如果已到最后一级：
  - 关闭 LED
  - 停止渐暗定时器
  - 回到 `Idle`

#### 事件 2：`EnvBrightEntered`
动作：

- 立即关闭 LED
- 停止渐暗定时器
- 设置 `auto_dark_trigger_armed = true`
- 回到 `Idle`

#### 事件 3：`MotionDetected`
动作：

- **完全忽略**

---

## 7.4 启动行为

系统上电启动时：

1. 初始化 PWM 为关闭
2. 初始化 BH1750
3. 连续读取 2 次亮度，建立初始 `env_dark`
4. 不因为“启动时刚好是暗环境”而自动亮灯

启动策略：

- 如果启动时已处于黑暗：
  - `env_dark = true`
  - `auto_dark_trigger_armed = false`
- 如果启动时处于明亮：
  - `env_dark = false`
  - `auto_dark_trigger_armed = true`

这样可以避免：

- 插上电源时因为房间本来就暗，设备立刻误开灯

---

## 8. 光照检测策略

## 8.1 BH1750 工作模式

采用：

- **连续高分辨率模式**

含义：

- BH1750 持续测量环境亮度
- MCU 每隔固定周期读取测量结果
- 不需要每次重新下发一次“开始测量”命令

该模式适合当前的 Type-C 常供电方案。

---

## 8.2 采样周期

- **每 1 秒读取一次 BH1750**

原因：

1. 主灯开/关响应足够自然
2. 调试更容易
3. 系统复杂度低
4. 不需要为了省电拉长周期

---

## 8.3 阈值设计

初始固定值：

- **开灯阈值：6 lux**
- **关灯阈值：25 lux**

### 规则

- `lux <= 6`：满足“进入暗环境”的候选条件
- `lux >= 25`：立即判定环境重新变亮
- `6 < lux < 25`：保持当前暗/亮状态，不切换

---

## 8.4 暗环境确认规则

为了避免短时遮挡、阴影、抖动导致误开灯：

- 连续 **2 次**采样都满足 `lux <= 6`
- 才认为环境真正进入暗环境

由于采样周期为 1 秒，因此：

- 最短确认时间约为 **2 秒**

这对于夜灯场景是合理的。

---

## 8.5 变亮确认规则

为了让“主灯打开 -> 夜灯立即关闭”足够自然：

- 只要 **1 次**采样满足 `lux >= 25`
- 就立即判定环境变亮

---

## 9. 人体事件处理策略

## 9.1 AM312 输入方式

- 使用 **GPIO 中断**
- 中断触发方式：**上升沿触发**

这样可以避免：

- 在高电平保持期间重复上报大量事件

---

## 9.2 ISR 行为

ISR 只做以下事情：

1. 记录一次人体事件
2. 向事件队列投递 `MotionDetected`
3. 尽快返回

**ISR 不做：**

- 不读 I2C
- 不计算状态机
- 不直接开灯/关灯
- 不打印复杂日志

---

## 9.3 AM312 事件的技术定位

本方案将 AM312 视为：

> “检测到人经过”的事件源

而不是依赖其高电平保持时间或精确脉宽。  
因此当前 MVP 不需要先实测其脉宽来决定主逻辑。

---

## 10. LED 与 PWM 控制

## 10.1 PWM 基本配置

建议：

- PWM 外设：ESP32-C3 的 LEDC
- 输出引脚：`GPIO5`
- PWM 频率：**4kHz**
- 分辨率：**10-bit**

这样可兼顾：

- 不可见闪烁
- 亮度控制足够细
- 实现简单

---

## 10.2 渐暗策略

总时长：

- **20 分钟**

亮度更新周期：

- **每 5 秒更新一次**

总步数：

- `20 * 60 / 5 = 240 步`

即：

- 从第 0 步到第 239 步
- 共 240 个亮度点
- 第 240 次结束后熄灯

---

## 10.3 渐暗曲线

不采用简单线性 PWM 下降，而采用：

- **近似人眼感知线性曲线**
- 实现方式：**查表**

### 计算方式建议

设：

- `N = 240`
- `max_duty = 1023`
- `p = 1.0 - step / (N - 1)`，表示感知亮度剩余比例
- `gamma = 2.2`

则：

```text
duty = round(max_duty * p^gamma)
```

特性：

- 开始阶段变化较平滑
- 后段不会显得突然变暗
- 更符合夜灯场景

---

## 10.4 LED 关闭方式

当满足以下任一条件时关闭 LED：

1. 渐暗 20 分钟结束
2. 环境变亮

关闭动作必须包括：

- 将 PWM duty 设为 0
- 停止 PWM 输出或保持低电平输出
- 停止渐暗定时器

---

## 11. 事件模型

建议定义以下事件类型：

```rust
enum Event {
    MotionDetected,
    EnvDarkEntered { lux: f32 },
    EnvBrightEntered { lux: f32 },
    FadeTick,
}
```

如果需要更详细调试，可扩展：

```rust
enum Event {
    MotionDetected,
    EnvDarkEntered { lux: f32 },
    EnvBrightEntered { lux: f32 },
    FadeTick,
    LuxSample { lux: f32 },
}
```

MVP 不建议让控制任务依赖 `LuxSample` 做逻辑，只建议用于日志。

## 13.2 关于 BH1750 驱动的建议

优先策略：

1. 优先寻找能直接配合 `embedded-hal` / `esp-idf-hal` 使用的成熟 BH1750 crate
2. 如果版本兼容性一般，则在项目内部封装一个**极薄的 service 层**
   - 只负责发送模式命令
   - 读取 2 字节数据
   - 换算 lux

这里不建议把它当成“完整自写驱动”，因为 BH1750 协议极简单，做一层 service 封装是合理的工程实现。

---

## 14. FreeRTOS 与事件队列实现建议

## 14.1 队列结构

建议建立一个全局事件队列：

- 队列长度：**16**
- 队列元素：`Event`

来源：

- AM312 ISR
- BH1750 采样任务
- 渐暗定时器回调

消费方：

- `controller_task`

---

## 14.2 为什么使用“单控制任务 + 多事件源”
这样做的优点是：

1. 状态机只有一个所有者，避免并发写冲突
2. 中断、定时器、采样逻辑解耦
3. 不需要多个任务同时直接操作 LED
4. 软件结构比“大循环轮询所有逻辑”更清晰

---

## 15. 控制逻辑细节

## 15.1 环境亮暗状态维护

建议在光照采样任务中维护：

- `env_dark`
- `dark_candidate_count`

逻辑：

```text
若当前 env_dark == false:
    lux <= 6  => dark_candidate_count += 1
    lux > 6   => dark_candidate_count = 0
    dark_candidate_count >= 2 => env_dark = true，发送 EnvDarkEntered

若当前 env_dark == true:
    lux >= 25 => env_dark = false，发送 EnvBrightEntered
    lux < 25  => 保持 env_dark = true
```

---

## 15.2 自动开灯资格

`auto_dark_trigger_armed` 的维护建议如下：

### 初始值
- 启动时如果环境明亮：`true`
- 启动时如果环境已暗：`false`

### 变亮时
- 设置为 `true`

### 由亮变暗自动开灯后
- 设置为 `false`

### 灯因 20 分钟结束而熄灭时
- 保持当前值不变，通常仍是 `false`

这样就能保证：

- 持续黑暗不会自动重复开灯
- 只有经历新的“亮 -> 暗”变化，才会再次自动开灯

---

## 15.3 黑暗中人体触发开灯

当收到 `MotionDetected` 时：

- 仅在 `state == Idle && env_dark == true` 时开灯
- 不依赖 `auto_dark_trigger_armed`
- 不改变“持续黑暗不可自动反复亮”的逻辑

这样就满足：

- 自动亮灯只响应真正的亮暗边沿
- 黑暗中人经过可以重新点亮

## 16. 板载 LED 处理方案

这是硬要求。

## 16.1 软件可控 LED
如果开发板带有软件可控 LED：

- 启动后立即配置为输出低电平
- 后续程序中不再使用该 GPIO
- 禁止任何库或示例代码对其做闪烁测试

## 16.2 不可控电源 LED
如果开发板上的 LED 是电源灯：

- 软件无法关闭
- 必须硬件处理

建议作为开发前的第一步检查项。

## 16.3 BH1750 模块的指示灯
原则完全相同：

- 有常亮灯就拆掉或遮光

否则在黑暗环境中，BH1750 模块自身就会成为干扰光源。

## 18. 启动流程

推荐启动顺序如下：


## 19. 简易时序说明

## 19.1 主灯关闭 -> 自动亮灯

```text
房间亮
  ↓
BH1750 周期采样
  ↓
连续2次 lux <= 6
  ↓
发送 EnvDarkEntered
  ↓
Controller 判断 auto_dark_trigger_armed == true
  ↓
LED 以 100% 亮度开启
  ↓
启动 5 秒 FadeTick 定时器
  ↓
进入 Lighting
```

---

## 19.2 主灯打开 -> 夜灯立即关闭

```text
夜灯处于 Lighting
  ↓
BH1750 周期采样
  ↓
任意1次 lux >= 25
  ↓
发送 EnvBrightEntered
  ↓
Controller 立即关灯
  ↓
停止 FadeTick 定时器
  ↓
回到 Idle
  ↓
auto_dark_trigger_armed = true
```

---

## 19.3 持续黑暗中，20 分钟结束

```text
夜灯处于 Lighting
  ↓
每5秒收到一次 FadeTick
  ↓
共执行 240 步
  ↓
PWM duty 降至 0
  ↓
关灯
  ↓
回到 Idle
  ↓
由于环境仍暗，auto_dark_trigger_armed 仍为 false
  ↓
不会再次自动亮灯
```

---

## 19.4 持续黑暗中，有人经过再次亮灯

```text
当前 Idle，且 env_dark == true
  ↓
AM312 上升沿中断
  ↓
ISR 投递 MotionDetected
  ↓
Controller 收到事件
  ↓
开灯
  ↓
启动 20 分钟渐暗
```

## 21. 推荐实现顺序

为了减少排障成本，建议按以下顺序开发：

### 阶段 1：硬件可用性验证
1. 检查开发板常亮 LED 是否可接受
2. 检查 BH1750 模块是否有常亮灯
3. 接通 LED + 电阻，验证 GPIO 可控亮灭
4. 验证 AM312 输出是否正常
5. 验证 BH1750 能稳定输出 lux

### 阶段 2：基础驱动验证
1. I2C 读 BH1750
2. PWM 驱动 LED
3. GPIO 中断接 AM312
4. 串口日志正常

### 阶段 3：事件系统
1. 建立队列
2. AM312 ISR 投递事件
3. 光照任务投递明暗事件
4. 控制任务可正常收事件

### 阶段 4：状态机
1. 完成 `Idle`
2. 完成 `Lighting`
3. 完成 `auto_dark_trigger_armed`

### 阶段 5：渐暗曲线
1. 先用简单 240 步线性表验证流程
2. 再切换成 gamma 表
3. 调整观感

### 阶段 6：产品联调
1. 主灯关 -> 自动亮
2. 主灯开 -> 自动灭
3. 黑暗中人经过 -> 重新亮
4. 持续黑暗不重复自动亮
5. 20 分钟自然熄灭

## 22. 必须关注的技术风险

## 22.1 开发板不可控常亮灯
这是当前最大的非软件风险。  
如果 ESP32-C3-Zero 板上有不可控电源 LED，则必须硬件处理。

## 22.2 BH1750 模块本身发光
模块电源灯会影响“暗环境”检测，也会破坏产品体验。必须去除。

## 22.3 暖白 LED 直驱亮度依赖型号
如果 5mm LED 的正向压降偏高，3.3V 直驱亮度可能偏低。  
当前方案接受这个前提，但需选型时注意。

## 22.4 AM312 触发保持时间
不同模块触发保持时间可能不同。  
本方案通过“上升沿事件化处理”降低对其具体脉宽的依赖。

## 23. 建议的最小验收标准

以下行为全部通过，即可认为 MVP 技术方案落地成功：

1. 上电后，板载可控 LED 不亮
2. BH1750 能稳定每秒输出 lux
3. AM312 触发可稳定产生人体事件
4. LED 能稳定由 PWM 控制亮度
5. 主灯关闭后约 2 秒内夜灯亮起
6. 主灯打开后约 1 秒内夜灯关闭
7. 夜灯亮后按 20 分钟渐暗
8. 20 分钟结束后灯完全熄灭
9. 持续黑暗中不会自动反复亮灯
10. 黑暗中人经过可重新亮灯

## 24. 推荐的软件实现基线

建议你把第一版代码目标控制在下面这个范围：

- 不做蓝牙
- 不做联网
- 不做参数存储
- 不做配置界面
- 不做复杂异常恢复
- 不做复杂多灯扩展
- 只完成：
  - 光照检测
  - 人体事件
  - PWM 渐暗
  - 状态机
  - 串口日志

这样最容易做成一版稳定 MVP。
